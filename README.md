\# Django Department Hierarchy API



Бэкенд-приложение на Django REST Framework для управления древовидной структурой департаментов организации. Проект содержит оптимизированные механизмы для предотвращения образования циклических зависимостей в иерархии должностей и подразделений.



\## 🚀 Основной функционал



\* \*\*Проверка иерархии на циклы\*\*: Эндпоинт для безопасного изменения родительского элемента с линейным обходом графа вверх. Возвращает статус `409 Conflict` в случае обнаружения петли.

\* \*\*Динамический контроль глубины\*\*: Возможность ограничения вложенности при обработке дерева (от 1 до 5 уровней).

\* \*\*Каскадные операции\*\*: Безопасное каскадное удаление узлов и переназначение связанных сотрудников.



\---



\## 🛠️ Спецификация API (OpenAPI / Swagger)



Полное описание эндпоинтов в формате OpenAPI 3.0 доступно в файле `openapi.yaml` в корневом каталоге проекта.



\### Ключевой эндпоинт: Проверка и изменение родителя



`POST /departments/{id}/check-hierarchy/`



\#### Параметры URL (Path)

\* `id` (integer, обязательно) — ID текущего департамента, у которого меняется родитель.



\#### Тело запроса (Body JSON)

```json

{

&#x20; "parent": 3,

&#x20; "max\_depth": 3

}

```



\#### Статус-коды ответов (Responses)

\* \*\*`200 OK`\*\*: Успешное обновление. Изменения применились к базе данных PostgreSQL, структура валидна.

\* \*\*`400 Bad Request`\*\*: Ошибка валидации (например, не передано обязательное поле `parent`).

\* \*\*`404 Not Found`\*\*: Целевой или новый родительский департамент не найден в базе данных.

\* \*\*`409 Conflict`\*\*: Обнаружен бесконечный цикл (попытка назначить департамент родителем самого себя или перенос вышестоящего узла внутрь дочернего).

\* \*\*`204 No\_content`\*\*: Успешное удаление. Изменения применены к базе данных PostgreSQL, структура валидна.

\## 💻 Локальное развертывание



\### 1. Клонирование и подготовка окружения

```bash

git clone <url-вашего-репозитория>

cd Ваш\_Проект

python -m venv venv

source venv/bin/activate  # Для Windows: venv\\Scripts\\activate

pip install -r requirements.txt

```



\### 2. Настройка базы данных

Убедитесь, что в файле `core/settings.py` в блоке `DATABASES` прописаны корректные доступы к вашей СУБД (PostgreSQL / MySQL / SQLite) \[4].



Примените миграции для автоматического создания таблиц `department` и `employee` в базе данных:

```bash

python manage.py makemigrations

python manage.py migrate

```

\### 3. Запустить приложение

Выполните в терминале (в папке проекта) команду:



```bash

docker-compose up --build

После запуска:

приложение будет доступно по адресу http://localhost:8000;

база данных PostgreSQL будет работать в отдельном контейнере;

данные PostgreSQL сохранятся в Docker‑томе postgres\_data.
